- Added a six-phase implementation plan because the current codebase is already partially strict but still anchored on `workflow.observers`, lacks `stdlib/` and `extensions/`, and therefore needs a staged kernel/runtime/optional-surface/workflow/parity rewrite instead of a single undifferentiated refactor.
- Captured the key non-obvious boundaries explicitly: narrow the canonical root export surface, replace observers rather than carrying two extension models, keep persisted `thread_id` session compatibility unless later clarified otherwise, and keep exact Autoloop-v1 session/log/git policy workflow-owned rather than leaking it into runtime or reusable extensions.
- PLAN-001 `blocking`: `plan.md` and `phase_plan.yaml` do not explicitly preserve or intentionally remove the current legacy config-file discovery behavior (`superloop.*` filenames) even though `runtime.config` currently exposes it and the verifier contract requires compatibility planning for configuration / CLI / developer-workflow surfaces. Failure scenario: phase 3 refactors `runtime.config` and the CLI, drops legacy discovery silently, and creates an unapproved break in existing repo or user config resolution. Minimal correction: add an explicit compatibility decision plus phase acceptance coverage stating whether legacy config discovery is preserved as generic runtime compatibility or intentionally removed with user-authorized migration notes.
- PLAN-002 `blocking`: The plan says “interface definitions” are included, but it does not pin the requested workflow-facing APIs for the optional extension surface, especially the git extension and session-path strategy. The request gives a concrete workflow opt-in shape (`GitTracking(policy=..., config=GitTrackingConfig(...))`) and requires a tiny extension surface, but `plan.md` only describes these modules conceptually. Failure scenario: implementation invents different names or runner-only wiring, drifts away from workflow-declared opt-in, or adds avoidable abstractions while still appearing to satisfy the high-level phases. Minimal correction: add explicit interface bullets and phase acceptance criteria for the workflow-facing git/session-path/tracing declarations and how they bind through `Workflow.extensions`.
- Revised the plan to close PLAN-001 and PLAN-002: `plan.md` now explicitly preserves `superloop.*` config discovery as a generic runtime compatibility boundary, and both `plan.md` plus `phase_plan.yaml` now pin the workflow-facing optional extension APIs (`GitTracking(...)`, `SessionPaths(...)`, `Tracing(...)`) together with their binding and test/acceptance coverage.
