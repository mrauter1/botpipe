# Plan ↔ Plan Verifier Feedback

- Replaced the empty plan with a five-phase implementation plan tied to actual repo coupling: public/simple surface cleanup, compiler/validation canonicalization, engine/provider/persistence cleanup, repo consumer migration plus optimizer separation, and final strictness/regression verification. This phase order matches where legacy names currently live (`autoloop/simple.py`, `core/*`, `runtime/static_graph.py`, loader/state handling, `stdlib`, and tests) and keeps resume-risk migration narrowly scoped to persisted run data.
- PLAN-001 `blocking`: The plan does not cover two explicit spec requirements: schema registry centralization and git-tracking ownership cleanup. Evidence: `plan.md` and `phase_plan.yaml` never mention the requested `core/schema_registry.py` consolidation or the requirement that git tracking move to runtime-owned policy instead of workflow declarations. Failure mode: an implementation can finish all planned phases yet still leave scattered schema IDs and dual workflow/runtime git-tracking ownership in place, violating the requested cleanup and preserving duplicated ownership. Minimal correction: add explicit milestone/phase scope, acceptance criteria, and validation for schema registry consolidation plus runtime-owned git-tracking API and workflow-surface removal.
- PLAN-002 `blocking`: Route/default-behavior semantics are underspecified even though they are a core requested behavior contract. Evidence: the spec requires explicit coverage for lowercase `"done"` as the plain-step default route, reserved terminal handling, control-route injection, and default route tables for `step`, `produce_verify_step`, `python_step`, `workflow_step`, and operation nodes; `plan.md` only says topology comes from step-local routes and declaration order. Failure mode: implementers can remove `flow`/`transitions` yet silently change `question`/`blocked`/`failed`, `accepted`/`needs_rework`, or operation-node behavior. Minimal correction: add explicit phase scope and acceptance criteria for canonical default-route injection and control-route invariants across every step kind.
- PLAN-003 `blocking`: The revised Pydantic state model is only partially planned and does not address item-scoped and step-item-scoped state handling or the “no half-working API” constraint. Evidence: `plan.md` covers workflow and step state renaming, but it does not say how `Worklist` item state, step-item state, related prompt references, checkpoint/resume behavior, or explicit deferment/removal will be handled. Failure mode: cleanup can ship with mixed descriptor/Pydantic state semantics or partially exposed item-state APIs, which the revised spec explicitly forbids. Minimal correction: add explicit scope for item/step-item state migration or an explicit plan to suppress any incomplete public surfaces while preserving checkpoint/prompt behavior.
- Updated the plan to address PLAN-001 through PLAN-003 by adding explicit scope for `core/schema_registry.py`, runtime-owned git tracking, canonical default-route/control-route behavior across all step kinds, and item/step-item state migration or suppression. The phase plan now carries those requirements in phase scope and acceptance criteria so implementation cannot accidentally skip them.
